# 📋 Инструкция по разработке — cognitive-core

> Этот файл содержит правила и принципы, которым необходимо следовать при работе над проектом.  
> Обращаться к нему перед каждым этапом разработки.

---

## 1. Следование плану

- Разработка ведётся **строго по текущему плану/roadmap'у**:
  - [`TODO.md`](TODO.md) — мастер-роадмап (P0→P1→P2→P3→архитектурные слои)
  - [`docs/ACTION_PLAN.md`](docs/ACTION_PLAN.md) — детальный план с code snippets и effort-оценками
- Перед началом нового этапа — **свериться с TODO.md** на предмет текущего прогресса и приоритетов.
- Принцип приоритизации: **сначала hardening, затем retrieval, затем расширение**.

---

## 2. Согласование с пользователем

- **План разработки** каждого этапа и его пунктов **согласовывается с пользователем** перед началом реализации.
- Не начинать реализацию без явного подтверждения плана пользователем.
- При возникновении неоднозначностей — спрашивать, а не предполагать.

---

## 3. Тестирование

- Запуск тестов (`pytest`, `mypy`, `ruff` и прочих) выполняется **только по указанию или одобрению пользователя**.
- Не запускать тесты автоматически после каждого изменения — только когда пользователь попросит или одобрит.
- При написании кода — учитывать совместимость с существующими тестами, но не запускать их самостоятельно.

---

## 4. Удаление файлов

- **Удаление файлов** (исходников, документации, тестов) — **только с явного разрешения пользователя**.
- Перед удалением — обосновать причину и получить подтверждение.

---

## 5. Стиль кода и документации

- Все комментарии и docstrings — **на русском языке**.
- Код следует стандартам проекта: `ruff` (B/SIM/C4/RET/PIE), `mypy` strict, `pytest` для тестов.
- Coverage gate: **70%** (текущий: 84.44%).

---

---

## 7. Правило принятия решений

При возникновении новой идеи — три вопроса:

1. Помогает ли это закрыть текущий этап?
2. Улучшает ли наблюдаемость, стабильность или retrieval?
3. Не уводит ли в research раньше времени?

**да / да / нет** → брать · **нет / нет / да** → откладывать в P3/Research
